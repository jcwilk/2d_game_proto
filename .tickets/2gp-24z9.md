# Desktop WASD merged with directional movement

Add keyboard **WASD** (and optionally **arrow keys**) as a second DOM-native input path: implement in **`src/input/directionalChrome.ts`** or a **sibling module under `src/input/`** (no barrel file required—import from the new module in `main.ts`). Normative alignment: **`specs/viewport-square-dpad-chrome.md`** **§6.3** (continuous velocity / opposite cancel / normalize onto `CHROME_MOVE_SPEED`), **§8** (keyboard mirrors the same movement state as chrome), plus existing **§6.2** pointer semantics for chrome only—keyboard must **not** use tap-latch; **merge** key state with `chrome.getActiveDirections()` via per-direction **boolean OR** (or `mergeActiveDirections(a,b)`) **before** `chromeMoveVelocityFromActiveDirections`. Ignore **`keydown` with `event.repeat === true`** for movement state. **Desktop gating:** expose an injectable predicate (e.g. `isKeyboardMoveEnabled`, defaulting to `matchMedia('(pointer: fine)')` or the project’s chosen heuristic) so Vitest can assert on/off without a real browser; document hybrid devices in a short code comment. **Lifecycle:** register `keydown`/`keyup` on `window` with paired **teardown** and hook **`import.meta.hot.dispose`** next to `chrome.detach()` so listeners do not leak. **Focus:** when focus is in a focusable control (e.g. merchant action buttons from **2gp-v4mi**), do not steal keys—gate or skip movement keys per `document.activeElement`. `preventDefault` only where needed to avoid page scroll. Update **`main.ts` `preupdate`** to use merged active directions. Tests: **`keyboardDirections.test.ts`** (or similar) for key state → `ActiveDirectionFlags` and merge with sample chrome flags; mock the desktop predicate.

## Acceptance Criteria

`npm run typecheck` && `npm test` pass. With dev server, holding W/A/S/D moves the avatar on a desktop-class viewport; d-pad still works. No movement from WASD when the injectable desktop predicate is false (unit-tested). `specs/viewport-square-dpad-chrome.md` velocity mapping unchanged aside from the merged key source. No listener leak on HMR dispose.
